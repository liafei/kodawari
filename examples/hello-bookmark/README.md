# Hello-bookmark: kodawari 5-minute demo

End-to-end walkable example: empty directory → working FastAPI service with
SQLite + tests, all generated by kodawari from this PRD.

Validated runtime: ~10 minutes wall-clock on `gpt-5.5` planner + `mimo-v2.5-pro`
executor/reviewer. Output: 5/5 tasks completed, 6/6 verify rounds with real
`pytest` invocations, 6/6 peer review rounds (one self-healing fix-loop on
the route layer task).

## What you'll need

- Python 3.11+
- One of:
  - Claude Code CLI authenticated (`claude auth login`) — **easiest, no API keys**
  - OR an OpenAI-compatible API endpoint + key
  - OR Anthropic API key
- 10–20 minutes of wall-clock for the autopilot to converge

## Walkthrough

```bash
# 1. Create a fresh project directory and copy this PRD
mkdir -p ~/playground/bookmark
cp PRD.md ~/playground/bookmark/PRD.md
cd ~/playground/bookmark

# 2. Generate models.yaml + .env.example with the wizard
kodawari init-wizard --preset claude-subscription --yes
#    Pick the preset that matches what you have:
#    --preset claude-subscription   → no API keys, uses claude CLI
#    --preset openai-compatible     → single OpenAI-compatible endpoint
#    --preset multi-provider        → distinct planner/reviewer/executor providers

# 3. If you picked openai-compatible or multi-provider, fill in keys
cp .env.example .env
# (edit .env, paste your API keys)

# 4. Sanity-check before burning tokens
kodawari doctor preflight --feature bootstrap --prd ./PRD.md
#    Expected: status PASS (or WARN with non-blocking hints)

# 5. Run the full pipeline (planner → executor → reviewer → release gate)
kodawari work-all \
  --feature bootstrap \
  --prd ./PRD.md \
  --planner-route model \
  --max-cycles 3 \
  --max-wall-clock-seconds 1800 \
  --real-peer-review
```

## What to expect

After ~10 minutes you'll see (rough sequence):

```
[plan]  prd-intake → architecture-plan → init → task-plan
        TASK_GRAPH.json with 4-5 tasks (schema/repo/service/route + tests)

[work]  per task: design → implement → verify (pytest!) → rules-gate → peer-review
        ~6 rounds per task on average

[work]  final status: AWAITING_DECISION (release gate stops for human approval)

[your-dir]/
├── app/
│   ├── __init__.py
│   ├── main.py             # FastAPI app + 4 endpoints
│   ├── service.py          # business logic with ValueError(...) on bad input
│   └── repository.py       # SQLite SQL + schema migration
├── tests/
│   └── test_api.py         # TestClient + happy/edge cases
├── planning/bootstrap/
│   ├── PRD_INTAKE.json
│   ├── ARCHITECTURE_PLAN.json
│   ├── TASK_GRAPH.json
│   ├── TASK_CARD_T1.json … TASK_CARD_T5.json
│   └── .autopilot_rounds.jsonl   # full audit trail
├── PRD.md
├── .claude/workflow/models.yaml
└── .env
```

## Going from `AWAITING_DECISION` to shipped

```bash
# Inspect what happened
kodawari status --feature bootstrap

# Run the tests one more time yourself (sanity)
cd ~/playground/bookmark
python -m pytest tests/ -q

# If happy, ship
kodawari decide --feature bootstrap --action accept
```

## What if it gets stuck

Look at the `.autopilot_rounds.jsonl` log:

```bash
# Show the last round
tail -1 planning/bootstrap/.autopilot_rounds.jsonl | jq .
```

Key signals:
- `stage_status: blocked` with `last_error: ...` → see the error
- `stall_counters.no_write_iterations > 10` → executor in a read-loop;
  consider switching to a stronger executor model
- `verify_check.command_executed: false` → tests didn't run; check
  `verify_targets` was resolved

See [docs/OPERATOR_RUNBOOK.md](../../docs/OPERATOR_RUNBOOK.md) for the
full error code index.
